# ⚡ Платформа управления энергопотреблением (Enterprise Energy) — Песочница для QA-автоматизаторов

Добро пожаловать в обучающий репозиторий для интернов и младших QA-инженеров! Этот проект представляет собой изолированную «песочницу» (sandbox), созданную для прокачки навыков **гибридного тестирования** (ручное тестирование + автоматизация) на современной микросервисной архитектуре.

Репозиторий содержит Enterprise-версию облачной SaaS-платформы, которая мониторит энергопотребление, использует JWT-авторизацию, генерирует счета на оплату и выгружает аналитические отчеты. Это безопасная среда, где можно изучить архитектуру фреймворков, подход BDD, работу с ORM-базами данных и CI/CD пайплайнами.

---

## 🏗️ Стек технологий

* **Бэкенд приложения:** Node.js, Express.js (модульный роутинг), Prisma ORM, СУБД SQLite, JWT (JSON Web Tokens) для защиты API.
* **Фронтенд приложения:** Single Page Application (SPA), HTML / JavaScript, Tailwind CSS (через CDN).
* **Фреймворк автоматизации:** JavaScript, Cypress (v10+), @badeball/cypress-cucumber-preprocessor (BDD).
* **Сборщик тестов:** esbuild (настроен через `@bahmutov/cypress-esbuild-preprocessor`).
* **CI/CD:** GitHub Actions (автоматический прогон автотестов при пуше).

## 🚀 Быстрый старт

### 1. Подготовка окружения
Убедись, что на твоем компьютере установлен [Node.js](https://nodejs.org/) (рекомендуется версия 18 или выше).

### 2. Установка зависимостей
Склонируй этот репозиторий, открой терминал в корневой папке проекта (`SaS Project`) и установи все пакеты одной командой:
```bash
npm install
```

### 3. Инициализация базы данных (Prisma)
Так как проект использует настоящую БД, перед первым запуском нужно накатить схему таблиц:
```bash
npx prisma db push
```
*(Это создаст локальный файл `dev.db` в папке `prisma` с таблицами пользователей и счетов).*

### 4. Запуск локального сервера приложения
Чтобы увидеть интерфейс платформы и дать возможность Cypress взаимодействовать с ним, запусти локальный бэкенд:
```bash
node server.js
```
После успешного запуска открой браузер и перейди по адресу: **`http://localhost:3000`**
* *Данные для входа администратора (по умолчанию):* Логин: `admin` / Пароль: `password123`

### 5. Запуск Cypress Test Runner
Открой **вторую вкладку терминала** (не закрывая первую, где работает сервер!) и выполни команду:
```bash
npx cypress open
```
* В открывшемся окне выбери **E2E Testing**.
* Выбери любой удобный браузер (например, Chrome).
* В списке тестов кликни по любому файлу с расширением `.feature` (например, `billing.feature`).

## 🎯 Практические задания и челленджи для интернов

Чтобы закрепить материал, выполни следующие задачи:

### Задание 1: Оптимизация авторизации (Уровень: JWT & API)
Изучи файл `cypress/support/commands.js`. Разберись, как работает кастомная команда `cy.loginByApi()`. В Enterprise-версии недостаточно просто положить данные в LocalStorage.
* *Задача:* Проверь, как Cypress перехватывает токен авторизации и передает его в заголовке `Authorization: Bearer <token>` для доступа к защищенным роутам телеметрии.

### Задание 2: Работа с файловой системой (Уровень: Скачивание артефактов)
На вкладке "Reports" появилась новая кнопка выгрузки CSV-отчета.
* *Задача:* Напиши автотест, который кликает на кнопку скачивания, а затем проверяет папку `cypress/downloads/`. Тест должен убедиться, что файл `report.csv` реально скачался и внутри него содержатся правильные заголовки таблиц.

### Задание 3: Укрощение асинхронного UI через интерцепты (Уровень: Сеть)
Дашборд загружает данные графиков асинхронно после переключения вкладок.
* *Задача:* Вместо использования нестабильных `cy.wait(5000)`, настрой `cy.intercept()` на роут `/api/telemetry`. Заставь Cypress дожидаться ответа от сервера, прежде чем проверять видимость элементов на странице.

### Задание 4: Интеграция с базой данных (Уровень: Prisma)
Cypress работает в изоляции от бэкенда, но иногда тестам нужны данные из БД.
* *Задача:* Попробуй написать скрипт внутри `cypress.config.js` (через `setupNodeEvents`), который делает запрос напрямую в SQLite базу через Prisma Client, чтобы перед тестом очищать таблицу счетов (Invoices).

## 💡 Важные правила автоматизатора
1. **Никаких жестких ожиданий по времени:** Никогда не используй явные задержки вроде `cy.wait(3000)`. Используй алиасы сети `cy.wait('@apiCall')` или встроенные проверки (`.should('be.visible')`).
2. **Gherkin — для бизнеса, а не для кода:** В `.feature` файлах не должно быть технических деталей. Пиши `When I click the Download button`, но **никогда** не пиши `When I click on element #btn-download`. Селекторы прячем в JS.
3. **Чистое состояние (Clean State):** Каждый тест должен быть независимым. Не полагайся на данные, оставшиеся от предыдущего сценария.
