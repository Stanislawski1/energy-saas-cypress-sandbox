# ⚡ Платформа управления энергопотреблением (Enterprise Energy) — Песочница для QA-автоматизаторов

Добро пожаловать в обучающий репозиторий для интернов и младших QA-инженеров! Этот проект представляет собой изолированную «песочницу» (sandbox), созданную для прокачки навыков **гибридного тестирования** (ручное тестирование + автоматизация).

Репозиторий содержит упрощенную копию корпоративной облачной SaaS-платформы, которая в реальном времени мониторит энергопотребление, анализирует данные «умных сетей» (Smart Grid) и автоматически генерирует счета на оплату (billing). Это безопасная среда, где можно изучить архитектуру современных тестовых фреймворков, подход BDD, перехват API-запросов и работу с тестовыми данными.

---

## 🏗️ Стек технологий

* **Бэкенд приложения:** Node.js, Express.js, СУБД SQLite (In-Memory, работает прямо в оперативной памяти).
* **Фронтенд приложения:** Чистый HTML / JavaScript / CSS.
* **Фреймворк автоматизации:** JavaScript, Cypress (v10+), @badeball/cypress-cucumber-preprocessor (BDD).
* **Сборщик тестов:** esbuild (настроен через `@bahmutov/cypress-esbuild-preprocessor` для мгновенного запуска).

## 🚀 Быстрый старт

### 1. Подготовка окружения
Убедись, что на твоем компьютере установлен [Node.js](https://nodejs.org/) (рекомендуется версия 18 или выше).

### 2. Установка зависимостей
Склонируй этот репозиторий, открой терминал в корневой папке проекта (`SaS Project`) и установи все пакеты одной командой:
```bash
npm install
```

### 3. Запуск локального сервера приложения
Чтобы увидеть интерфейс платформы и дать возможность Cypress взаимодействовать с ним, необходимо запустить локальный бэкенд:
```bash
node server.js
```
После успешного запуска открой браузер и перейди по адресу: **`http://localhost:3000`**
* *Данные для входа администратора (по умолчанию):* Логин: `admin` / Пароль: `password123`

### 4. Запуск Cypress Test Runner
Открой **вторую вкладку терминала** (не закрывая первую, где работает сервер!) и выполни команду:
```bash
npx cypress open
```
* В открывшемся окне выбери **E2E Testing**.
* Выбери любой удобный браузер (например, Chrome).
* В списке тестов кликни по любому файлу с расширением `.feature` (например, `billing.feature`), чтобы запустить автоматические сценарии.

## 🎯 Практические задания и челленджи для интернов

Чтобы закрепить материал и понять, как устроена автоматизация на реальных проектах, выполни следующие задачи:

### Задание 1: Оптимизация авторизации (Уровень: API)
Изучи файл `cypress/support/commands.js`. Разберись, как работает кастомная команда `cy.loginByApi()`. Она позволяет обойти медленную UI-страницу логина, отправляя прямой запрос на бэкенд и подкладывая токен авторизации сразу в `LocalStorage` браузера.
* *Задача:* Напиши новый тест-кейс, который использует эту команду для мгновенного перехода в защищенную зону дашборда, минуя форму ввода.

### Задание 2: Укрощение асинхронного UI через интерцепты (Уровень: Сеть)
Дашборд энергоплатформы загружает данные асинхронно. Вместо использования нестабильных "тайм-аутов вслепую" вроде `cy.wait(5000)`, мы учимся слушать сетевой слой.
* *Задача:* Открой файл `cypress/e2e/step_definitions/billingSteps.js`. Разберись, как конструкция `cy.intercept('POST', '/api/billing/generate').as('generateBillApi')` ловит отправку счета. Попробуй самостоятельно перехватить `GET` запрос, который подтягивает графики телеметрии.

### Задание 3: Написание BDD-сценария с нуля (Уровень: Бизнес-логика)
* *Задача:* Открой файл `cypress/e2e/features/login.feature` (сейчас он пуст или не завершен). Напиши сценарий негативного тест-кейса на понятном английском/русском языке Gherkin (например, попытка входа с неверным паролем). После этого реализуй соответствующие JS-функции в папке `step_definitions/`.

## 💡 Важные правила автоматизатора
1. **Никаких жестких ожиданий по времени:** Никогда не используй явные задержки вроде `cy.wait(3000)` для ожидания элементов. Используй `cy.wait('@имя_алиаса_сеть')` или встроенные проверки Cypress (`.should('be.visible')`).
2. **Gherkin — для бизнеса, а не для кода:** В `.feature` файлах не должно быть технических деталей. Пиши `When I click the submit button`, но **никогда** не пиши `When I click on element #btn-submit-id-v2`. Все селекторы должны прятаться внутри JS-кода.
3. **Переиспользуемость:** Старайся писать шаги (Step Definitions) так, чтобы один и тот же шаг (например, проверка открытия страницы) можно было использовать в пяти разных тестах.
